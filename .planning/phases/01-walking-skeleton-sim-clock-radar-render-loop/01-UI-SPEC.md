# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Stack note:** This project is a native Python + pygame-ce **desktop** application — there is no React/Next.js/Vite, no npm ecosystem, no DOM/CSS, and no shadcn component registry anywhere in this stack (confirmed: no `components.json`, no `package.json` in the repo). The shadcn initialization gate in the standard UI research process is **not applicable** and was skipped for that reason, not because it was declined. All visual elements in this phase are drawn via direct `pygame.draw.*` / `pygame.font` calls onto a `Surface`. This UI-SPEC therefore expresses the design contract in radar/vector-display terms (canvas geometry, draw-call styling, color, sim-tick-driven motion) rather than web layout/breakpoint terms.

### Radar Canvas Geometry (Phase 1's actual spacing surface)

These are the load-bearing "spacing" values for this phase — pixel geometry for the radar canvas and its drawn elements, taken as the canonical contract (matches `.planning/phases/01-walking-skeleton-sim-clock-radar-render-loop/01-RESEARCH.md` Code Examples exactly, so research, plan, and implementation all reference the same numbers).

| Token | Value | Usage |
|-------|-------|-------|
| Window size | 1280 × 800 px, fixed | Application window and canvas dimensions (D-03 — no resize/scaling logic in Phase 1) |
| Radar center | (640, 400) px | Screen-space origin for range rings, sector lines, and heading-vector angle math |
| Range ring spacing | 80 px | Radial distance between consecutive range rings (multiple of 4) |
| Range ring count | 4 | Number of concentric rings drawn |
| Range ring stroke | 1 px, `pygame.draw.circle(..., width=1)` | Ring line weight |
| Sector line count | 8 | Radial reference lines at 45° intervals |
| Sector line style | `pygame.draw.aaline` (anti-aliased, 1px) | Radiates from radar center to the outermost ring's radius |
| Aircraft dot radius | 5 px | `pygame.draw.circle` radius — **exception** to the 4px grid (a touch/legibility-driven exception, same category as the 44px touch-target exception pattern) |
| Heading vector length | 40 px | Line length from aircraft position along `heading_deg` (multiple of 4) |
| Trail length | 8 points | `collections.deque(maxlen=8)`, one point appended per **sim tick** (not per render frame) — ≈4 seconds of history at the 2Hz tick rate |
| Trail dot radius | `max(1, int(5 * 0.4 * fade))` px, `fade` = age-based linear falloff | Oldest trail points shrink toward 1px; newest approach ~2px — exception to the 4px grid (sub-pixel-scale decorative element) |

| Destructive | not applicable in Phase 1 | No destructive actions and no alert states exist in this phase's scope (STCA-style conflict alerts arrive in Phase 5 — define a destructive/alert red then, not now) |

Accent reserved for: the aircraft dot, its heading vector line, and its trail dots — nothing else in Phase 1 uses this color family.

## Rendering Behavior Notes

Contract details beyond the standard template dimensions, carried over from `01-RESEARCH.md` and `01-CONTEXT.md` because they are visual/interaction-affecting and executor-critical:

- **Window is fixed, not resizable** (D-03) — `pygame.display.set_mode((1280, 800))` with no resize event handling needed this phase.
- **Heading convention:** 0° = up/north, increasing clockwise (standard compass/radar convention) — `end = (x + sin(rad)*len, y - cos(rad)*len)`. All later phases (datablocks, vectoring instructions) must use this same convention; do not silently switch to math-standard (0°=east, counter-clockwise) anywhere in the codebase.
- **Static background is pre-rendered once**, not redrawn every frame — range rings and sector lines are blitted from a cached `Surface` (`build_static_background`), only the aircraft dot/vector/trail redraw per frame.
- **Trail points are sourced once per sim tick**, never once per render frame — append inside the `on_tick` callback, not inside the draw loop. This keeps the trail representing true sim-tick history (a ~4s visual trace) rather than a dense cluster of near-duplicate interpolated positions.
- **Position/heading are interpolated for rendering** (previous/current sim-tick snapshot, `alpha = accumulator / tick_dt`) so the aircraft appears to glide smoothly across render frames between the 2Hz sim ticks, per CORE-03/RADAR-03. The interpolation itself must never mutate the authoritative `Aircraft` state — it only reads two frozen snapshots and returns a third, throwaway value.
- **Wrap-at-edge exception (D-02):** when a sim tick causes the aircraft to wrap from one canvas edge to the opposite edge, do **not** linearly interpolate position across that tick's render frames — this would draw a fast streak across the entire canvas. Detect the wrap (large jump between `prev.x`/`curr.x` or `prev.y`/`curr.y`, e.g. greater than half the canvas width/height) and snap directly to the post-wrap position for that tick's frames instead.
- **Aircraft dot draws last**, on top of its own heading vector and trail, so it is never visually occluded by its own trail history.
